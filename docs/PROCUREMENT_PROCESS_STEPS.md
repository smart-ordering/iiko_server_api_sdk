# Подробное описание процесса: закупки, приемка, документы и финансы

Ниже — подробный пошаговый сценарий (step-by-step) с разделением по фронту/беку, данным и iiko API.
Файл предназначен как базовое ТЗ для реализации DDD-проекта.

---

## 0. Инициализация и подключение к iiko

### Цель
Создать базу справочников и синхронизировать ключевые сущности: склады/цеха, товары, поставщиков, остатки.

### Фронт
1. Экран подключения интеграции iiko (ввод URL, ключей, проверка доступа).
2. Выбор филиалов/складов, которые участвуют в закупках.
3. Просмотр справочников (товары, поставщики, склады).

### Бек
1. Настроить клиента iiko и хранение токенов/ключей.
2. Загрузить справочники и сохранить в локальную БД.
3. Завести ревизии синхронизации и расписание обновлений.

### iiko API
- Департаменты чейны оргнащиации склады 
- Справочники: `GET /v2/entities/list`
- Товары: `GET /v2/entities/products/list`
- Поставщики: `GET /suppliers`
- Прайс-листы: `GET /suppliers/{code}/pricelist`
- Остатки: `POST /inventory` или `GET /v2/reports/balance/stores`

### Данные (хранение)
- `suppliers`, `supplier_pricelists`
- `products`, `product_categories`, `units`
- `stores`, `departments`
- `stock_balances`, `sync_revisions`

## 1. Первичные заявки от поваров/барменов

### Цель
Собрать потребность по складу/цеху на основе факта и рекомендаций.

### Фронт
1. Форма заявки по складу/цеху.
2. Отображение остатков, списаний, рекомендаций.
3. Сохранение черновика и отправка заявки.

### Бек
1. Создать агрегат `PurchaseRequest`.
2. Валидировать лимиты и минимальные уровни.
3. Привязать заявку к складу/филиалу и пользователю.

### iiko API
- Остатки: `POST /inventory` или `GET /v2/reports/balance/stores`
- Списания/расходы (опционально): `POST /v2/reports/olap`

### Данные (хранение)
- `purchase_requests`, `purchase_request_items`
- `limit_rules`, `stock_snapshots`

---

## 2. Формирование закупочных листов (главный закупщик)

### Цель
Сконсолидировать заявки по складам/филиалам и подготовить закупочные листы.

### Фронт
1. Список закупочных листов по складам/цехам.
2. Редактирование количества/цены.
3. Выбор поставщика и даты поставки.

### Бек
1. Создать агрегат `PurchaseList`.
2. Рассчитать суммы с учетом минималок и лимитов.
3. Привязать поставщика и окна поставок.

### iiko API
- Поставщики: `GET /suppliers`, `GET /suppliers/{code}/pricelist`
- Товары: `GET /v2/entities/products/list`

### Данные (хранение)
- `purchase_lists`, `purchase_list_items`
- `supplier_candidates`, `delivery_windows`

---

## 3. Финансовое утверждение закупки

### Цель
Получить итоговое одобрение закупки от доверенного финансового лица.

### Фронт
1. Экран консолидации закупки.
2. Кнопки “Утвердить/Отклонить”.

### Бек
1. Доменный шаг `Approval`.
2. Фиксация решения и аудит.

### iiko API
- Нет прямого эндпойнта (локальная бизнес-логика).

### Данные (хранение)
- `approvals`, `approval_decisions`, `approval_audit`

---

## 4. Генерация заявок поставщикам + WhatsApp

### Цель
Автоматически сформировать заявки по поставщикам и отправить в WhatsApp.

### Фронт
1. Статусы отправки/принятия/контрпредложения.
2. Таймер ожидания ответа.

### Бек
1. Агрегат `SupplierOrder`.
2. Генерация QR и токена заявки.
3. Idempotency и outbox для предотвращения дублей.

### iiko API
- Нет прямого эндпойнта (внешняя интеграция WhatsApp).

### Данные (хранение)
- `supplier_orders`, `supplier_order_items`
- `supplier_order_tokens`, `outbox_messages`, `idempotency_keys`

---

## 5. Приемка товара на месте

### Цель
Фиксировать факт поставки, расхождения и фото-доказательства.

### Фронт
1. Скан QR, открытие заявки.
2. Сверка накладной, взвешивание, фото.
3. Фиксация статуса: “Все принято / Расхождения / Отказ”.

### Бек
1. Агрегат `Acceptance`.
2. Проверки критериев приемки (вес, количество, качество, производитель).
3. Хранение фотодоказательств.

### iiko API
- Нет прямого эндпойнта (пока локальная фиксация).

### Данные (хранение)
- `deliveries`, `acceptances`, `acceptance_items`
- `acceptance_photos`, `discrepancies`

---

## 6. Подтверждение приемки доверенным лицом

### Цель
Проверить корректность приемки и утвердить ее.

### Фронт
1. Экран проверки приемки.
2. Кнопка “Одобрить приемку”.

### Бек
1. Смена статуса `Acceptance`.
2. Логирование утверждения.

### iiko API
- Нет прямого эндпойнта.

### Данные (хранение)
- `acceptance_approvals`

---

## 7. Пересчет нетто/брутто и финальная накладная

### Цель
Пересчитать фактические веса и сформировать финальный документ для iiko.

### Фронт
1. Экран пересчета нетто/брутто.
2. Предпросмотр финальной накладной.

### Бек
1. Применить таблицы конверсий фасовок.
2. Сформировать `FinalInvoice`.
3. Выгрузить документ в iiko.

### iiko API
- `POST /documents/import/incomingInvoice`
- `POST /documents/import/returnedInvoice` (для возвратов)
- `POST /documents/unprocess/incomingInvoice` (если нужен откат)

### Данные (хранение)
- `conversion_tables`, `conversion_rules`
- `final_invoices`, `invoice_lines`, `iiko_document_links`

## 8. Калькуляции и фасовки

### Цель
Обновлять фасовки и калькуляции, влияющие на расчеты нетто/брутто.

### Фронт
1. Форма заявки на новую фасовку.
2. Экран статусов работы калькуляторов.

### Бек
1. Агрегат `PackagingChangeRequest`.
2. Обновление таблиц конверсий и (опционально) iiko справочников.

### iiko API (опционально)
- `POST /v2/entities/productScales/save|update`
- `GET /v2/assemblyCharts/getAll`

### Данные (хранение)
- `packaging_requests`, `packaging_changes`, `product_scales_cache`

---

## 9. Финансовый модуль: выплаты и отчеты

### Цель
Управлять оплатами поставщикам и контролировать лимиты.

### Фронт
1. Список выплат, статусы “Оплачен/НЕ оплачен”.
2. Фильтры по датам/поставщикам/филиалам/категориям.

### Бек
1. Агрегаты `Payment`, `Limit`, `ReportSnapshot`.
2. Привязка платежных документов.

### iiko API
- Балансы: `GET /v2/reports/balance/counteragents`
- Остатки по складам: `GET /v2/reports/balance/stores`
- Документы: `GET /documents/export/outgoingInvoice`

### Данные (хранение)
- `payments`, `payment_files`, `payment_statuses`
- `branch_limits`, `report_snapshots`

---

## 10. Сквозные DDD-контексты (сводно)

### Bounded Contexts
- Master Data (справочники)
- Procurement (заявки, закупочные листы, лимиты)
- Supplier Orders (заказы поставщикам)
- Receiving (приемка)
- Inventory Documents (накладные и выгрузка в iiko)
- Finance (выплаты и отчеты)

### Общие правила
- Все справочники хранят `external_id` из iiko.
- Все интеграционные операции — через `idempotency_keys`.
- Фото и документы хранятся в object storage, в БД только метаданные.
